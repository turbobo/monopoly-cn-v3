## 大富翁中国行 v3 — 技术架构文档

**版本**: 3.1.0 | **更新日期**: 2026-06-12 | **项目路径**: monopoly-cn-v3

---

### 一、项目概览

大富翁中国行 v3 是一款基于 Web 的多人在线大富翁桌游，支持三种游戏模式：AI对战、本地同屏、在线对战。游戏以中国城市为棋盘主题（厦门到北京的28格），使用 Canvas 2D 渲染棋盘与动画，GoEasy PubSub 实现实时联机通信，部署为纯静态站点（Next.js Static Export + EdgeOne Pages CDN）。

项目从 v1（单机大富翁精简版）经历 v2（Socket.IO 在线版）迭代而来。v3 的核心目标是在纯前端静态部署的前提下实现免费、低延迟的在线多人对战，不依赖任何自建后端服务器。

---

### 二、技术栈

**前端框架**: Next.js 14.2 (App Router) + React 18.3 + TypeScript 5.4，构建输出为静态 HTML（`output: 'export'`），所有页面在 `src/app/page.tsx` 中用 `'use client'` 客户端组件实现。

**样式**: Tailwind CSS 3.4 + 自定义 CSS 动画（fadeIn、bounce-in、shake、pulse），字体使用 Google Fonts 的 Noto Sans SC。

**渲染**: 原生 Canvas 2D API（`board-renderer.ts`），不使用任何图形库。棋盘 28 格采用 8.5×8.5 网格布局，四角为 1.3 倍大格，每格通过 `getTilePosition()` 计算绝对坐标。

**音效**: Web Audio API 程序化合成（`sound.ts`），不加载任何音频文件。包含骰子翻滚（白噪声带通脉冲序列）、骰子落地（低频撞击+高频碎裂）、逐格跳动（正弦衰减）、购买（C-E-G 三音叮咚）、支付（三角波下降）、破产（锯齿波警示）、玩家加入（上升三连音）、玩家离开（下降三连音）八种音效。

**联机通信**: GoEasy PubSub SDK 2.14.9（`goeasy-manager.ts`），免费额度 500 DAU / 10万条消息每月。使用杭州节点（hangzhou.goeasy.io）。

**部署**: EdgeOne Pages 静态站点，`edgeone.json` 配置 `outputDirectory: "out"`，构建命令 `npm run build`，Node 版本 20.18.0。

#### 4.1 游戏引擎（game-engine.ts）

引擎是一个纯函数状态机，不包含任何副作用。核心数据结构和函数：

**棋盘**: `BOARD` 数组共 28 格（7×4 边），顺时针排列。每格有类型（property/railroad/chance/tax/start/jail/parking/goto_jail/utility）、价格、三级租金数组、颜色分组。颜色分组通过 `COLOR_GROUPS` 映射表定义 7 个颜色组（棕色2块、浅蓝1块、橙红3块、粉色1块、绿色3块、蓝色3块、紫色2块），其中浅蓝（大连）和粉色（杭州）是独立单色组（仅1块地皮，无同色加成）。同色组地皮齐全后租金按 `rent[2]`（全套租金）计算，拥有≥2块但未齐全时按 `rent[1]` 计算。

**GameState 状态结构**: 包含 `players`（Player 数组）、`currentPlayer`（当前玩家索引）、`round`/`maxRounds`（回合计数）、`dice`（双骰值）、`phase`（roll/action/auction/trade/end 五阶段）、`log`（消息日志）、`gameOver`/`winner`、`difficulty`（三档难度影响 AI 决策系数）、`roadblocks`（路障数组）、`priceHikes`（涨价卡数组，含 tileId/ownerPlayerId/roundsLeft）、`lastCardRound`（上次发卡的回合）。

**Player 扩展字段**: `cards`（GameCard 数组，当前持有的道具卡）、`freePassActive`（免费卡是否激活）。

**回合推进 `executeTurn()`**: 接受 GameState 和可选的预掷骰子，返回消息数组。处理流程：破产跳过 → 监狱逻辑（掷双数越狱 / 3回合保释金50 / AI主动支付保释金） → 掷骰移动 → 经过起点奖金（基础200，每5回合加50，上限400） → 路障检查（踩中则跳过本回合） → 格子效果 → 破产检查（自动卖地6折回收并清空地皮列表） → 下一玩家。格子效果通过 `processTile()` 函数处理，非 AI 玩家落在无人地皮上时 `phase` 切到 `action` 等待购买决策。税收：个人所得税固定 ¥100，房产税固定 ¥150。

**AI 系统**: 三种性格（aggressive/balanced/conservative）配合难度系数（easy×1.5 / normal×1.0 / hard×0.7）。购买决策基于 `money / price` 倍数和性格阈值；拍卖决策按性格计算最高出价（60%~95% 地皮价）；交易决策按 1.2~1.5 倍价格接受；变卖资产优先卖非整组最便宜地皮；主动交易寻找缺失同色组的地皮并报价。

**AI 进阶策略**:
- **同色组进度加成**：已拥有同色组 ≥50% 时购买意愿大幅提升（系数降至 0.5~0.6），已有 ≥1 块时也有加成（系数 0.85）
- **回合数加成**：游戏进入中后期（>10回合）AI 更激进（系数 0.9），>20回合后更激进（系数 0.8）
- **主动变卖资产**：资金低于阈值时（激进¥50/平衡¥100/保守¥150），优先卖出不在同色组内的最便宜地皮
- **主动发起交易**：AI 会主动寻找能补齐同色组的地皮，向其他玩家出价购买（激进×1.6/平衡×1.4/保守×1.3）
- **监狱出狱策略**：激进型第1回合主动支付保释金、平衡型第2回合支付、保守型等到第3回合强制出狱

**道具卡系统**: 每5回合（第5、10、15...回合）所有未破产玩家各获得1张随机道具卡。五种卡类型：
- **遥控骰子**（🎯）：指定本次掷骰点数（2-12），通过 `useRemoteDice()` 将总点数拆分为两个骰子值
- **路障卡**（🚧）：在指定格子放置路障，其他玩家踩中后本次移动无效，通过 `checkRoadblock()` 检测
- **交换卡**（🔄）：与任意未破产玩家互换棋盘位置，通过 `useSwapCard()` 交换
- **免费卡**（🛡️）：激活后下次需付租金时免除费用，通过 `freePassActive` 标记
- **涨价卡**（📈）：自己的一块地皮租金翻倍，持续3个游戏回合，通过 `tickPriceHikes()` 每回合递减

#### 4.2 棋盘渲染器（board-renderer.ts）

`BoardRenderer` 类封装了全部 Canvas 2D 渲染逻辑，以 60fps（requestAnimationFrame）驱动。

**坐标系**: `resize()` 根据容器尺寸和 DPR（上限2）计算画布大小，存储 `this.dpr` 用于全局缩放。提供 `px(n)` 和 `font(size, family, weight)` helper 方法，将所有像素值和字体大小转换为 CSS 像素（乘以 dpr），确保不同 DPR 屏幕上视觉一致。`tileSize = size / 8.5`，四角 `cornerSize = tileSize × 1.3`。28 格的 `getTilePosition()` 返回每格的绝对像素坐标和尺寸。

**渲染层次**（从底到顶）：深蓝径向渐变背景 → 对角线织纹（0.025 透明度） → 顶部柔光 → 格子卡片（圆角矩形，有主时高亮紫色 / 有主时半透明白 / 无主时更淡） → 格子内容（emoji + 城市名 + 价格或主头像） → 路障/涨价卡标记 → 中心标题"大富翁中国行" → 骰子动画 → 玩家棋子 → 浮动文字 → 粒子。

**道具卡地图标记**:
- **路障**（🚧）：格子背景变为橙红色调（rgba(255,100,50,0.2)），右上角显示🚧角标
- **涨价**（📈）：格子背景变为金黄色调（rgba(255,200,50,0.2)），左上角显示📈角标

**公开方法**: 除渲染方法外，还暴露 `isMoving()` 检查移动动画状态、`hitTest(clientX, clientY)` 将屏幕坐标转换为格子索引、`getTileScreenCenter(index)` 获取格子 CSS 像素中心坐标、`showCenterFloat(text, color)` 显示居中浮动文字，供 `page.tsx` 的地皮信息弹窗功能调用。

**骰子动画**: 三阶段式——上升翻滚（0~0.6 进度，骰子分开并旋转） → 悬停切换面值（0.6~0.85，快速随机切换点数） → 下落落地（0.85~1.0，抛物线加速下落带拖尾粒子）。落地后有多段弹跳（上弹 → 压扁 → 微抖），落地光环扩散，结果数字弹出（缩放+淡入+胶囊背景）。骰子本体用线性渐变模拟光照，点数颜色区分 1/4 为红色。

**逐格移动动画**: 棋子沿棋盘边缘跳跃弧线（正弦抛物线），水平移动用缓入缓出（quadratic ease-in-out），每格触发 `onStep` 回调（音效+粒子），经过的格子留下半透明 ♟️ 路径标记。棋子本身带 squash & stretch 形变（起跳纵向拉伸，落地横向压扁）和整圈旋转，♟️ 标记反向旋转保持正立。

**玩家棋子**: 静态棋子分两级大小（当前玩家 18px / 其他 12px），当前玩家有发光光圈、脉冲外圈、顶部箭头指示器。棋子本体用径向渐变（高光→主色→暗部）+ 白色描边 + emoji 头像 + 底部标签（名字胶囊+现金标签，颜色随余额变化：>500绿/>200黄/其他红）。

#### 4.3 音效引擎（sound.ts）

全部使用 Web Audio API 程序化合成，零音频文件依赖。`AudioContext` 单例懒初始化，支持 `suspended` 状态自动 `resume()`。全局静音开关 `muted` 控制所有音效。

骰子翻滚 `playDiceRoll()` 生成 8 段白噪声短脉冲（30ms），每段经带通滤波器（2000~5000Hz 随机）和指数衰减包络，模拟骰子在桌面滚动的声音，间隔 70~100ms 随机排列。

骰子落地 `playDiceLand()` 由两部分叠加：低频正弦波从 180Hz 快速滑降到 60Hz（撞击感），加上高通滤波（3000Hz）的白噪声脉冲（碎裂感）。

逐格跳动 `playStepSound()` 是 600~800Hz 随机正弦波 80ms 衰减到 300Hz。购买 `playBuySound()` 是 C5-E5-G5 三音琶音（80ms 间隔）。支付 `playPaySound()` 是 400→150Hz 三角波下降。破产 `playBankruptSound()` 是三个递降锯齿波（220/180/140Hz）。

玩家加入 `playPlayerJoinSound()` 是 A4→C#5→E5 上升三连音（大三和弦琶音），营造欢迎氛围。玩家离开 `playPlayerLeaveSound()` 是 C5→G4→E4 下降三连音，营造告别氛围。

#### 4.4 联机管理器（goeasy-manager.ts）

`GoEasyManager` 类封装了 GoEasy SDK 的 PubSub 通信，接口设计兼容原 PeerJS 管理器（`initialize`/`createRoom`/`connectToRoom`/`broadcast`/`sendToPeer`/`onMessage`/`onDisconnection`/`onConnection`/`onConnectionStatusChange`/`destroy`/`leaveRoom`/`getRoomId`/`getClientId`/`getIsHost`/`setIsHost`/`startHeartbeat`），方便上层代码无缝切换。

**SDK 集成**: 使用静态 `import GoEasySDK from 'goeasy'`（`@ts-ignore` 绕过类型检查）。`init()` 是全局单例方法，通过 `static sdkInitialized` 标记防止重复初始化。连接参数：`host: 'hangzhou.goeasy.io'`, `appkey: BC-849dcab428524333a394bbc19744e72b`, `modules: ['pubsub']`。客户端 ID 格式为 `${playerName}-${6位随机ID}`。

**房间模型**: 一个房间对应一个 GoEasy 频道，频道名格式 `monopoly-${roomId}`。创建房间 = 订阅新频道；加入房间 = 订阅已有频道。房间号是 6 位去歧义字母数字（排除 I/O/0/1）。

**消息协议**: 所有消息统一为 `PeerMessage` 接口——`{ type, payload, from, timestamp }`。`type` 取值：`player-join`（加入房间）、`room-info`（房间信息广播）、`player-action`（玩家操作：roll/buy/card）、`game-state`（完整游戏状态同步）、`dice-rolled`（骰子结果+动画同步）、`player-leave`（离开房间）、`card-action`（道具卡操作）、`sync-request`（断线重连后 Guest 请求同步最新状态）、`chat`（聊天消息）、`ping`/`pong`（心跳）、`error`（房间满员等错误通知）。消息体 JSON 序列化后通过 `pubsub.publish()` 发送到频道。收到的消息通过 `handleIncomingMessage()` 解析，自动过滤自己发的消息（`data.from === clientId`），自动响应 ping 回复 pong，然后分发给注册的 messageHandlers。

**注意**: `sendToPeer()` 在 GoEasy PubSub 模式下实际是广播（无法点对点），接收方通过消息中的 `target` 或 `playerName` 字段过滤。

**心跳与断线重连**: 游戏开始后由 `startHeartbeat()` 启动三个定时器——每 15 秒发送 ping 消息，每 20 秒检查所有对端的 `peerLastSeen` 时间戳（超过 45 秒未收到任何消息判定离线），每 30 秒检查 GoEasy 连接状态（断连时自动调用 `tryReconnect()`）。重连流程：断开 → `disconnect()` 清理旧连接 → 500ms 延迟 → 重新 `connect()` → 重新 subscribe 频道 → 重发消息队列（最多缓存 50 条非心跳消息）。重连最多重试 **5 次**（`MAX_RECONNECT_ATTEMPTS`），每次失败后 5 秒延迟再试。重连成功后 Guest 自动发送 `sync-request` 从 Host 获取最新游戏状态。连接状态变化通过 `ConnectionStatus`（`reconnecting`/`connected`/`failed`）回调通知 UI 层。

**生命周期**: `destroy()` 清理所有资源——停止心跳、取消订阅频道、断开连接、清空 handler 数组、重置 `sdkInitialized`（允许新实例重新 init）、标记 `destroyed` 防止僵尸回调。页面 `beforeunload` 事件和 React unmount 回调自动触发 destroy。`createRoom()`/`connectToRoom()` 内置 15 秒超时和防重复调用保护。`leaveRoom()` 中先广播 `player-leave` 再延迟 500ms 销毁，确保离开消息送达。

### 五、在线对战架构

#### 5.1 通信拓扑

```
  Host (房主)                    Guest (客机)
  ┌──────────┐                   ┌──────────┐
  │ GoEasy   │◄─── Channel ────►│ GoEasy   │
  │ Manager  │   monopoly-XXX   │ Manager  │
  └────┬─────┘                   └────┬─────┘
       │                              │
       │ publish / subscribe          │ publish / subscribe
       │                              │
       ▼                              ▼
  ┌─────────────────────────────────────────┐
  │        GoEasy Server (杭州节点)          │
  │        WebSocket PubSub 消息中继         │
  └─────────────────────────────────────────┘
```

所有玩家通过 WebSocket 连接到 GoEasy 杭州服务器，同一个房间的双方订阅相同频道 `monopoly-${roomId}`。消息通过 GoEasy 服务器中继广播，不使用 P2P 直连（区别于之前的 PeerJS WebRTC 方案）。

#### 5.2 Host-Authority 模式

游戏逻辑采用 Host-Authority（房主权威）架构：

**房主（Host）** 负责所有游戏计算——掷骰子、执行回合、计算租金、判断破产、推进回合。每次计算完成后将完整的 `GameState` 通过 `game-state` 消息广播给所有客机。

**客机（Guest）** 仅发送 `player-action` 消息（roll/buy）给房主，不执行任何游戏逻辑。收到房主的 `game-state` 后更新本地状态和 UI。

这个设计保证了游戏状态的一致性，避免了双端计算可能产生的分歧。

```
  │◄── onMessage(dice-rolled) ────────│                        │
  │                                    │                        │
  │  [播放骰子动画 + 移动动画]           │                        │
  │                                    │                        │
  │                                    │◄── publish(game-state) ──│
  │◄── onMessage(game-state) ─────────│                        │
  │                                    │                        │
  │  [更新游戏状态 + UI]                │                        │
```

#### 5.4 动画同步

**房主端**: `executeHostRoll()` 使用 `gameRef.current`（避免闭包过期）计算回合结果，**先广播 `dice-rolled`**（含骰子值、玩家索引、起始位置、计算后的完整 GameState），**再本地播放动画**。广播和动画并行，减少客机延迟。通过 `animatingRef` 守卫防止动画期间重复执行（网络延迟可能导致重复 action）。

**客机端**: 收到 `dice-rolled` 后调用 `playGuestDiceRolled()` 播放骰子动画（2x 加速）和移动动画（1.8x 加速），补偿网络延迟，使两端体验接近同步。动画完成后用 `gameRef.current` 获取最新状态（可能已被 `game-state` handler 更新为更新的版本），统一通过 `setGame()` 更新 React 状态。

**动画冲突保护**: 如果动画播放期间收到新的 `dice-rolled`（如无需购买时房主快速推进回合），不会直接覆盖当前动画，而是将新消息追加到 `pendingDiceRolledRef`（数组队列）。当前动画回调完成后检查队列，延迟 100ms 后补播下一个动画，确保每个回合的移动动画都完整呈现。补播动画使用更高加速倍率（骰子 3x，移动 2.5x）以追赶进度。

**`game-state` 缓存**: 如果 `game-state` 在动画完成前到达（`animatingRef.current === true`），仅缓存到 `gameRef.current` / `messagesRef.current`，不触发 React re-render，由 `dice-rolled` 动画回调统一应用最新状态。

#### 5.5 购买决策流程

**在线购买时序**: 
1. 房主 `executeTurn()` 返回 `phase === 'action'` → 广播 `dice-rolled`（含待购买状态）
2. 客机动画完成后检查 `latestGame.phase === 'action'`，若是自己回合则显示 `buyPrompt`
3. 客机发送 `player-action: { type: 'buy', buy: true/false }` 给房主
4. 房主执行购买/跳过 → `finalizeTurn()` 推进回合 → 广播 `game-state`

**购买超时**: 房主在动画完成后为非自己的购买决策设置 **20 秒超时**（需覆盖客机动画时间 ~5s + 网络延迟 ~2s + 用户思考时间）。超时后自动"放弃购买"并推进回合。玩家断线时也会清除购买超时。

**`handleBuy` 防护**: 统一使用 `gameRef.current` 读取最新状态（避免 React state 闭包过期），操作完成后立即同步 `gameRef.current = newState`。防双击守卫使用 `buyPromptRef.current`（ref 模式）而非 `buyPrompt`（state），避免 `useCallback` 闭包捕获旧值导致按钮失效。

#### 5.6 断线检测与容错

**心跳**: 游戏开始后每 15 秒发送 ping，每 20 秒检查一次对端活跃度，45 秒无消息判定断线。断线触发 `disconnectionHandler`，房主侧清除购买超时、标记断线玩家破产、回收资产（6折变卖地皮）、推进回合、广播状态更新。客机侧显示"与房主连接断开"并更新玩家列表。

**房间存在性检测**: 客机加入后 **10 秒**内未收到 `room-info` 消息则判定房间不存在或房主离线。超时回调中校验 `peerRef.current === joinedPeer`（捕获的实例引用），防止用户退出再重新加入时误销毁新实例。

**资源清理**: 每次创建/加入房间前先 `destroy()` 旧的 GoEasyManager 实例，防止僵尸连接。页面 `beforeunload` 和 React unmount 双重保障清理。`leaveRoom()` 延迟 500ms 销毁，确保 `player-leave` 消息送达。

### 六、UI 与状态机

`page.tsx` 是一个单文件 React 组件（2269 行），包含完整的状态管理和 UI 渲染。

**Screen 状态机**:

```
menu ──(开始游戏)──► setup ──(本地/AI)──► game ──(结束)──► end
                         │                    ▲              │
                         │                    │              │
                         ├──(在线:创建)──► lobby ──(开始)───►│
                         │                    │              │
                         └──(在线:加入)──► lobby ────────────►│
                                                              │
                    menu ◄────(再来一局/返回大厅)──────────────┘
```

5 个 Screen 状态：`menu`（主菜单标题） → `setup`（模式选择/参数设置/在线大厅） → `lobby`（在线等待房间） → `game`（游戏进行中） → `end`（结算画面）。

**状态管理策略**: 使用 26 个 `useState` 管理 UI 状态，21 个 `useRef` 同步最新值以避免闭包陈旧问题（`gameRef`/`messagesRef`/`onlinePlayersRef`/`myNameRef`/`buyTimeoutRef`/`aiTimeoutRef`/`guestRollTimeoutRef`/`playersRef`/`screenRef`/`animatingRef`/`roomValidatedRef`/`buyPromptRef`/`pendingDiceRolledRef`/`executeHostRollRef`/`forcedDiceRef`/`handleLocalRollRef`/`processAITurnsRef`/`canvasRef`/`rendererRef`/`peerRef`/`logRef`）。所有 ref 通过 `useEffect` 与对应 state 保持同步。关键函数 `executeHostRoll` 通过 ref 间接引用，确保 `player-action` handler 中调用时始终拿到最新闭包。

**在线模式工具函数** (`online-utils.ts`):
- **`slimGame(gs)`**: 精简 GameState 用于网络传输，移除 `log` 字段（日志单独传输），减少消息体积
- **`trimMessages(msgs)`**: 只保留最近 20 条消息，防止日志无限增长
- **`mergeMessages(localLog, remoteMsgs)`**: Guest 端合并本地历史日志与远程消息，保持日志连续性

**布局**: 响应式 Flex 布局，桌面端左侧棋盘（Canvas）+ 右侧信息面板（320px），移动端上下排列（棋盘占 50dvh，面板占剩余空间）。Canvas 尺寸在 resize/orientationchange 事件时重算（含 100ms 防抖定时器），使用 `touch-none` 禁止触摸默认行为。支持棋盘点击/触摸查看地皮信息弹窗（`hitTest` + `tileInfo` 状态），以及道具卡面板（`showCardPanel` 状态控制展开/收起）。

### 九、关键设计决策

**为什么选择 Host-Authority 而非同步计算？** 大富翁的游戏逻辑涉及随机数（骰子、机会卡）和状态判断（破产、卖地），如果双端各自计算，微小的时序差异就会导致状态分叉。Host-Authority 让所有计算集中在房主端，客机只做 UI 渲染，从根本上消除了状态不一致的可能。代价是客机有几十毫秒的操作延迟，但对回合制游戏来说完全可以接受。

**为什么用 PubSub 而非 WebSocket 长连接？** GoEasy 的 PubSub 模型天然适合房间制游戏——一个频道就是一个房间，广播消息给所有订阅者。不需要自己管理连接池和消息路由。免费额度 10 万条消息/月对 4 人游戏绰绰有余（每局约 200~500 条消息）。

**为什么 Canvas 而非 DOM 渲染棋盘？** Canvas 提供了更精细的动画控制（逐帧插值、粒子系统、形变效果），性能也更好（一次 draw 调用 vs 数百个 DOM 节点更新）。60fps 的渲染循环让骰子翻滚和棋子跳跃动画流畅自然。

**为什么程序化合成音效而非加载音频文件？** Web Audio API 合成音效零网络请求、零文件依赖、极小体积。所有音效代码在 `sound.ts` 中 232 行实现 8 种音效，不需要加载任何 mp3/wav 文件。对于这种轻量级桌游，程序化音效的品质足够好，且不会增加首屏加载时间。

**为什么静态导出而非 SSR？** 大富翁是纯客户端游戏，所有计算都在浏览器完成，不需要服务器端渲染或 API 路由。静态导出（`output: 'export'`）生成的 HTML/CSS/JS 可以直接部署到任何 CDN（EdgeOne Pages、Cloudflare Pages、Vercel 等），零运维成本，全球 CDN 加速，且完全免费。

### 十、瓶颈分析与优化方案

当前技术栈在免费条件下已跑通完整链路，但存在若干明确瓶颈。以下按优先级排列优化方案，所有方案均保持全免费或极低成本。

#### 10.1 联机稳定性：GoEasy → Supabase Realtime

**现状**：GoEasy 免费版 500 DAU + 10 万消息/月。4 人玩一局约 200-500 条消息，DAU 到 100 人就可能触及限额。

**优化方案**：迁移到 Supabase Realtime。

| 对比项 | GoEasy（当前） | Supabase Realtime |
|--------|--------------|-------------------|
| 用户上限 | 500 DAU | 5 万 MAU |
| 消息限额 | 10 万/月 | 无限 |
| 数据库 | 无 | 500MB PostgreSQL |
| 用户认证 | 无 | 免费 Auth（邮箱/手机） |
| 国内延迟 | 低（杭州节点） | 中（海外节点，约 200-400ms） |

迁移工作量：`GoEasyManager` 已做好接口封装（initialize / createRoom / broadcast / onMessage），只需写一个 `SupabaseManager` 实现相同接口，`page.tsx` 无需改动。大富翁是回合制游戏，几百毫秒延迟完全不影响体验。

附带收益：500MB 数据库可做游戏存档、战绩统计、排行榜。

#### 10.2 渲染质量：裸 Canvas → Phaser 3

**现状**：`board-renderer.ts` 900+ 行手写 Canvas，实现了基础绘制和动画（骰子翻滚、棋子逐格跳动、squash & stretch 形变），并内置 DPR 缩放支持。

**优化方案**：迁移到 Phaser 3 游戏引擎。

- 精灵图动画：棋子移动更丝滑，支持帧动画和骨骼动画
- 粒子系统：掷骰烟雾、金币飞舞、购买庆祝特效
- 摄像机：支持棋盘缩放和平移，大屏/小屏自适应
- 音频管理：内置音频管理器，替代 `sound.ts` 的手写 Web Audio 代码
- 碰撞检测：内置物理引擎，方便后续扩展（如道具系统）

代价：包体增加约 1MB（gzip 后 300KB），需要重写 `board-renderer.ts`。但代码总量会大幅减少（Phaser 内置了大量功能）。建议作为 v4 版本的重构方向。

#### 10.3 游戏存档：无 → Supabase / localStorage

**现状**：关闭页面游戏数据就丢失了，不支持中途退出恢复。

**方案 A（推荐）：Supabase 免费数据库**

- 新增 `game_saves` 表：存 GameState JSON + 房间号 + 时间戳
- 支持断线重连恢复棋局（重进房间自动加载存档）
- 附带排行榜、战绩统计等社交功能
- 免费 500MB 数据库存几十万局数据绰绰有余

**方案 B（零成本快速方案）：localStorage**

- 每次 `setGame` 后同步写 `localStorage`
- 页面加载时检查是否有未完成的游戏 → 自动恢复
- 改动量极小（半天完成），缺点是只支持单机/本地模式，换设备丢失

#### 10.4 AI 对手：规则策略 → DashScope 大模型

**现状**：AI 对手已实现较完善的规则策略系统——三种性格（aggressive/balanced/conservative）配合难度系数，含同色组进度加成、回合后期激进加成、主动变卖资产、主动发起交易、完整道具卡 AI 决策（`aiUseCardDecision`）、拍卖/交易策略、监狱出狱策略。但规则 AI 的行为模式固定，高手玩家容易预测。

**优化方案**：接入 DashScope Qwen API（免费 7000 万 Token）。

- 每次 AI 决策传入：当前资金、地皮分布、对手状态、可购买地皮信息
- AI 返回：买/不买 + 理由（可显示为"AI 思考中..."气泡）
- 每次决策约 200 token，一局 30 回合 4 个 AI 玩家 ≈ 2.4 万 token
- 免费额度能玩约 **300 万局**，完全无成本压力
- 可分难度等级：简单（规则决策）、普通（AI 建议）、困难（AI 深度推理）

#### 10.5 域名：免费子域 → 自有域名

**现状**：ClouDNS 免费子域名（`game-online.playai.cloud-ip.cc`），不利于传播和记忆。

**优化方案**：Cloudflare Registrar 购买 `.xyz` 域名，成本价约 **7 元/年**（如 `monopoly-abc.xyz`）。DNS 天然在 Cloudflare，EdgeOne Pages 绑定后访问体验好很多。这是唯一的硬成本，但极低。

#### 10.6 PWA 离线支持

**现状**：每次都需要从浏览器打开 URL，不能像 App 一样从桌面启动。

**优化方案**：

- 添加 `manifest.json`（应用名、图标、启动画面、主题色）
- 添加 Service Worker（缓存静态资源，支持离线打开）
- 用户可以"添加到主屏幕"，体验接近原生 App
- 纯前端改动，零成本，半天完成

#### 10.7 优化优先级总结

| 优先级 | 优化项 | 改动量 | 成本 | 核心收益 |
|--------|--------|--------|------|---------|
| **P0** | 游戏存档（localStorage） | 小（半天） | 0 元 | 解决关页面丢失问题 |
| **P0** | AI 对手升级（DashScope） | 小（1 天） | 0 元 | AI 体验质变，免费 300 万局 |
| **P1** | 联机升级（Supabase） | 中（2 天） | 0 元 | 500 DAU → 5 万 MAU |
| **P1** | PWA 离线支持 | 小（半天） | 0 元 | 类 App 体验，可装桌面 |
| **P2** | 域名 | 配置 | 7 元/年 | 传播友好，易于记忆 |
| **P3** | 渲染升级（Phaser 3） | 大（1 周） | 0 元 | v4 重构方向 |
